> **版本**: v1.0

## 1. 产品概述

### 1.1 产品定义

**AI Panel Studio** 是一款 AI 驱动的圆桌讨论演播厅 Web 应用。用户输入任意讨论话题并指定专家人数（默认 4 人），系统调用大模型动态生成主持人与专家嘉宾（姓名、职业/Title、立场、专属颜色）。用户确认阵容后进入演播厅，观看一场 AI 驱动、实时推进的圆桌讨论。

### 1.2 核心价值

- **零门槛发起讨论**：只需一个话题，AI 自动构建完整演播厅
- **沉浸式观看体验**：演播厅视觉风格，流式发言，实时状态
- **去中心化 Agent 讨论**：嘉宾自主决定发言时机，非机械轮替，接近真实讨论动态
- **结构化知识输出**：实时提炼共识与分歧，讨论结束生成总结报告

### 1.3 目标用户

- 想从多角度理解复杂话题的求知者
- 需要模拟观点碰撞的产品/策略人员
- 对 AI Agent 协作感兴趣的开发者/研究者

## 2. 功能需求

### 2.1 首页：讨论列表

| ID | 功能 | 描述 |
|----|------|------|
| F1.1 | 进行中讨论列表 | 展示所有状态为"进行中"的讨论，用户可点击加入观看 |
| F1.2 | 已结束讨论列表 | 展示历史讨论，用户可查看总结报告和完整 Transcript |
| F1.3 | 发起新讨论 | 入口按钮，进入话题配置与嘉宾生成流程 |
| F1.4 | 讨论状态标识 | 每个讨论卡片显示话题、嘉宾数、状态、创建时间 |

### 2.2 嘉宾生成

| ID | 功能 | 描述 |
|----|------|------|
| F2.1 | 话题输入 | 用户输入讨论话题（文本，建议最长 200 字） |
| F2.2 | 专家人数设定 | 用户指定专家人数（默认 4，范围 2-8） |
| F2.3 | LLM 生成阵容 | 调用 LLM 生成 1 位主持人 + N 位专家。每人包含：姓名、职业/Title、立场描述、头像提示词、专属颜色 |
| F2.4 | 编辑嘉宾信息 | 用户可修改任意嘉宾的姓名、Title、立场、颜色 |
| F2.5 | 单体重生 | 用户可对某一位嘉宾单独触发重新生成（不影响其他人） |
| F2.6 | 全体重生 | 用户可整体重新生成全部阵容 |
| F2.7 | 确认阵容 | 用户确认后进入演播厅，不可再编辑 |

### 2.3 演播厅：核心讨论

#### 2.3.1 主持人

| ID | 功能 | 描述 |
|----|------|------|
| F3.1 | 开场白 | 讨论开始时，主持人简短介绍话题与嘉宾 |
| F3.2 | 提问与串场 | 主持人在合适时机提出问题或引导话题转换 |
| F3.3 | 追问介入 | 主持人可对当前讨论内容进行追问 |
| F3.4 | 总结发言 | 讨论结束时，主持人用自然语言总结核心观点、共识与分歧 |
| F3.5 | 发言竞争 | 主持人与专家一样计算发言欲望值。欲望值等于专家时主持人优先发言；低于专家时该专家发言 |

#### 2.3.2 专家

| ID | 功能 | 描述 |
|----|------|------|
| F4.1 | 自主发言 | 专家根据当前 Transcript 自主决定发言（举手/抢答/补充/反驳），每次 1-2 句 |
| F4.2 | 禁止机械轮替 | 发言顺序由欲望值机制决定，不按固定顺序轮替 |
| F4.3 | 发言欲望值 | 每位专家每个发言周期计算欲望值（由统一标准制定），值高者获得发言权 |
| F4.4 | 决断链 | 欲望值 → 关注点时间距离 → 随机。主持人同分时优先于专家 |
| F4.5 | 专家数量 | 2-8 人（用户指定，默认 4） |

#### 2.3.3 独立观察员

| ID | 功能 | 描述 |
|----|------|------|
| F5.1 | 实时提炼 | 每有新发言，独立观察员立即判断是否产生新的共识或分歧 |
| F5.2 | 增量分析 | 观察员对每条发言做判断：该发言对已有观点产生了共识还是分歧 |
| F5.3 | 独立区域展示 | 共识/分歧列表在演播厅独立区域持续更新，不等讨论结束 |
| F5.4 | 置信度 | 每条共识/分歧带有观察员置信度（0-1） |

#### 2.3.4 专家状态小窗

| ID | 功能 | 描述 |
|----|------|------|
| F6.1 | 独立小窗 | 每位专家拥有独立状态显示区域 |
| F6.2 | 状态显示 | 三种状态：待机 (idle) / 准备发言 (preparing) / 发言中 (speaking)，实时切换 |
| F6.3 | 公开思考摘要 | 每位专家展示当前关注点或公开思考摘要（非隐藏 CoT） |
| F6.4 | 禁止暴露内部逻辑 | 页面任何位置不得出现 JSON、格式化字符、隐藏 CoT |

#### 2.3.5 Transcript（发言记录）

| ID | 功能 | 描述 |
|----|------|------|
| F7.1 | 发言人信息 | 每条发言展示：发言人姓名、Title、专属颜色标识 |
| F7.2 | 按时间序排列 | 发言按实际发表时间排列 |
| F7.3 | 屏蔽内部动作 | "举手"、"思考中"等内部调度动作不进入 Transcript |
| F7.4 | 流式显示 | 发言以流式（打字机效果）逐字呈现 |

#### 2.3.6 讨论控制

| ID | 功能 | 描述 |
|----|------|------|
| F8.1 | 用户推进 | 用户可手动触发下一轮发言 |
| F8.2 | 暂停/继续 | 用户可随时暂停或继续讨论 |
| F8.3 | 结束讨论 | 用户可强制结束讨论 |
| F8.4 | 自动结束 | 满足以下任一条件时主持人可决定结束：①达到预设最大轮次 ②连续无新共识/分歧产生 |

### 2.4 讨论总结报告

| ID | 功能 | 描述 |
|----|------|------|
| F9.1 | 完整 Transcript | 按发言序记录所有发言（含发言人信息） |
| F9.2 | 共识汇总 | 讨论中达成的所有共识 |
| F9.3 | 分歧汇总 | 讨论中产生的所有分歧（含是否已化解） |
| F9.4 | 主持人总结 | 主持人自然语言总结（作为最后一条发言） |
| F9.5 | 可读格式 | 纯文本/富文本展示，无 JSON 或格式化字符 |

## 3. 非功能需求

### 3.1 响应式布局

| 屏幕宽度 | 布局 | 说明 |
|----------|------|------|
| ≥ 1400px | 三栏 | 专家状态（左）+ Transcript（中）+ 共识/分歧（右），各栏独立滚动 |
| 800 - 1399px | 两栏 | 合并其中一栏 |
| < 800px | 单栏 | 垂直堆叠，各区域独立滚动 |

### 3.2 语言

- 全中文 UI（按钮、标签、提示、错误信息等）
- LLM 生成的讨论内容使用中文

### 3.3 性能

- WebSocket 流式事件延迟：发言 token 推送间隔不超过 100ms
- 首屏加载时间：< 3 秒

### 3.4 并发隔离

- 多个讨论可同时进行，各自状态、事件流、Transcript、共识/分歧完全隔离

### 3.5 安全

- DeepSeek API Key 仅在后端环境变量读取，不暴露给前端
- 无用户认证（纯匿名），讨论创建者通过 Session 标识

## 4. 交互与视觉设计（DDD 设计驱动阶段产物）

本节由 UI UX Pro Max 在 DDD 阶段详细输出，当前仅列出高层方向：

### 4.1 演播厅视觉风格

- **主题**：演播厅 / 电视圆桌节目风格
- **色调**：深色背景为主（类似演播厅灯光暗下），嘉宾颜色作为视觉高亮
- **动效**：发言时嘉宾卡片亮起、Transcript 流式打字机效果、状态指示灯变化

### 4.2 关键交互

- 嘉宾卡片以颜色标识，形成视觉归属感
- Transcript 区域以发言气泡或时间线展示
- 共识/分歧区域使用不同色彩标识（如绿色=共识，橙色=分歧）

## 5. 技术架构

### 5.1 架构模式

**Agent-Mediator 模式**：每个角色（主持人、专家、观察员）是独立 Agent，持有 Transcript 只读副本，自主计算发言欲望值。后端 Scheduler 负责冲突仲裁与调度。

### 5.2 技术栈

| 层 | 技术 |
|----|------|
| 前端 | Vite + React + TypeScript |
| 后端 | FastAPI (Python) + uv 包管理 |
| LLM | DeepSeek API (`deepseek-chat`, `https://api.deepseek.com/v1`) |
| 数据库 | SQLite |
| 实时通信 | WebSocket |
| UI 设计 | UI UX Pro Max（DDD 阶段） |

### 5.3 数据模型概览

核心实体：Discussion → PanelMember → Utterance → ConsensusDisagreement / ExpertStatusLog

（详细 DDL 在 SDD 阶段输出。）

### 5.4 WebSocket 事件模型（细粒度流式）

| 事件类型 | 方向 | 描述 |
|----------|------|------|
| `expert_status` | S→C | 某位嘉宾状态变更 + 关注点摘要 |
| `utterance_token` | S→C | 当前发言的增量 token（打字机效果） |
| `utterance_complete` | S→C | 当前发言完成（含完整元数据） |
| `consensus_update` | S→C | 新增或更新共识/分歧记录 |
| `discussion_paused` | S→C | 讨论暂停通知 |
| `discussion_resumed` | S→C | 讨论继续通知 |
| `discussion_ended` | S→C | 讨论结束通知 |
| `advance` | C→S | 用户手动推进下一轮 |
| `pause` | C→S | 用户暂停讨论 |
| `resume` | C→S | 用户继续讨论 |
| `end` | C→S | 用户强制结束 |

## 6. 工程化方法论

本项目采用 **多范式融合工程化架构**，分三阶段推进：

| 阶段 | 方法 | 核心产出 |
|------|------|----------|
| **前期** | SDD (Spec/Schema-Driven) | 数据模型 DDL、API 契约 OpenAPI 规范、WebSocket 事件 Schema |
| **中期** | DDD (Design-Driven) | UI UX Pro Max 输出完整 UI 设计、组件树、状态流转、交互规范 |
| **后期** | TDD (Test-Driven) | 单元测试（Agent 调度、欲望值计算、共识提炼）、集成测试（WebSocket 流）、E2E 测试（完整讨论流程） |

## 7. 用户角色与权限

| 角色 | 权限 |
|------|------|
| **讨论发起者** | 创建讨论、编辑嘉宾阵容、开始/暂停/继续/结束讨论、手动推进发言。拥有绝对控制权 |
| **观看者** | 加入已有讨论观看（实时或已结束），无法干预讨论进程 |

用户不参与讨论内容，仅为话题发起者和演播厅所有者。

| # | 决策点 | 选择 | 说明 |
|---|--------|------|------|
| 1 | 讨论推进 | 用户推进 + 固定轮次 + 无新共识时自动结束 | 三种结束条件组合 |

| 5 | LLM 调度 | 欲望值竞争机制 | 去中心化，决断链：值→时间→随机 |
| 6 | 主持人介入 | 需计算欲望值，同分优先 | 非绝对优先 |
| 7 | WebSocket | 细粒度流式事件 | 逐 token 推送 + 状态变更 |
| 8 | 用户身份 | 纯匿名 + 发起者绝对权限 | 无注册登录 |
| 9 | 持久化 | 文本存档 + 总结报告 | 讨论结束生成 |
| 10 | 架构 | Agent-Mediator 模式 | 独立 Agent + 统一调度器 |
